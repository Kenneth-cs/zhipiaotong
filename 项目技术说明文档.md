## 一、项目简介

**智票通**是一个面向中小企业的**发票智能识别与管理 SaaS 平台**。核心功能是：

- 用户上传 PDF 或图片格式发票（支持批量）
- 后端自动 OCR 识别 + AI 解析，提取结构化财务字段
- 去重检测、入库存档
- 前端展示历史记录、支持导出

**技术栈**：React 18 + TypeScript（前端）、Node.js + Express + TypeScript（后端）、MySQL（数据库）、火山引擎 OCRPdf + Ark 大模型（AI 能力）

| tax | DECIMAL(12,2) | 税额 |

## 五、发票识别核心流程（`server/volcengine.ts`）

### 当前架构：两阶段串行

```
用户上传文件
     │
     ▼
[Step 1] 火山引擎 OCRPdf（智能文档解析）
         Action: OCRPdf, Version: 2021-08-23, version: v3
         耗时约 1000~1500ms
         返回带版面结构的 Markdown 字符串
     │
     ▼
[Step 2] callLLMParse（大模型解析）
         模型: deepseek-v3-2-251201
         API: https://ark.cn-beijing.volces.com/api/v3/responses
         耗时约 5000~8000ms
         输入: OCRPdf Markdown 文本
         输出: 结构化 JSON
```

> **总耗时约 6~10 秒/张**。后续优化方向：实现可靠的正则快速解析路径，大部分发票可在 ~1.5s 内完成。

### 关键函数说明

| 函数 | 位置 | 说明 |
|------|------|------|
| `buildSignedRequest` | volcengine.ts | 实现火山引擎 V4 签名（HMAC-SHA256），用于 OCRPdf 请求鉴权 |
| `callSmartDocumentParse` | volcengine.ts | 调用 OCRPdf API，返回 Markdown |
| `callLLMParse` | volcengine.ts | 调用 Ark 大模型解析 Markdown 为结构化 JSON |
| `buildSubType` | volcengine.ts | 根据发票代码/类型推断 `invoiceSubType` 枚举 |
| `recognizeInvoice` | volcengine.ts | **主入口**，串联 OCRPdf → 大模型两步 |
| `acquireSemaphore` / `releaseSemaphore` | routes/ocr.ts | 信号量并发控制，默认最多 20 个并发 OCR 请求 |

### `invoiceSubType` 枚举含义

| 值 | 说明 |
|----|------|
| `FULLY_ELECTRONIC` | 数电发票（全电，无发票代码） |
| `VAT_SPECIAL` | 增值税专用发票 |
| `VAT_ELECTRONIC` | 增值税普通电子发票 |
| `REFINED_OIL` | 成品油发票 |
| `PAPER_GENERAL` | 纸质普通发票 |
| `UNKNOWN` | 未能识别类型 |

## 六、火山引擎 API 配置

### OCRPdf（智能文档解析）

- **Base URL**：`https://visual.volcengineapi.com`
- **Action**：`OCRPdf`
- **Version**：`2021-08-23`
- **鉴权**：火山引擎 V4 签名（`buildSignedRequest` 函数）
- **请求体**（`application/x-www-form-urlencoded`）：
  ```
  image_base64=<base64编码的文件内容>&version=v3
  ```
- **返回**：`{ code: 10000, data: { markdown: "...", detail: "..." } }`

### Ark 大模型（兜底解析）

- **Endpoint**：`https://ark.cn-beijing.volces.com/api/v3/responses`
- **Model**：`deepseek-v3-2-251201`
- **鉴权**：`Authorization: Bearer <ARK_API_KEY>`（在 `volcengine.ts` 中硬编码，后续应移至 `.env`）

### 环境变量（`.env`）

```env
VOLC_ACCESS_KEY_ID=<火山引擎 Access Key>
VOLC_SECRET_ACCESS_KEY=<火山引擎 Secret Key>
VOLC_OCR_CONCURRENT=20          # OCR 最大并发数
VOLC_OCR_MIN_INTERVAL=0         # 请求最小间隔(ms)，0=不限
```

## 七、前端架构

### 路由

```
/login      → Login.tsx（未登录默认跳转）
/           → Workspace.tsx（发票上传+识别）
/history    → History.tsx（历史记录）
```

### 上传队列（`src/lib/queue.ts`）

前端实现了上传队列，支持批量上传时按序/并行提交，防止界面卡顿。

### API 封装（`src/lib/api.ts`）

统一封装了所有 `fetch` 请求，自动携带 JWT token（存储在 `localStorage`），统一处理 401 跳转登录。

### 代理配置（`vite.config.ts`）

开发模式下，`/api` 前缀请求通过 Vite 代理到 `http://localhost:3001`（后端端口）。

## 九、当前进度与已知问题

### 已完成 ✅

- [x] 用户注册/登录（JWT 鉴权）
- [x] 发票文件上传（PDF + 图片）
- [x] 火山引擎 OCRPdf 集成，返回结构化 Markdown（~1s）
- [x] Ark 大模型（DeepSeek-V3）解析 Markdown → 结构化 JSON（~6-8s）
- [x] 去重检测（同用户同发票号码不重复入库，标记 `duplicate`）
- [x] 发票数据入库、历史记录查询
- [x] 信号量并发控制（默认 20 并发）
- [x] 混合解析策略框架（正则快速路径 + 大模型兜底，框架已就绪）

### 待解决 / 优化方向 🔧

1. **正则解析 `parseInvoiceMarkdown` 精度问题**（当前已禁用）
   - `buyerTaxId` / `sellerTaxId` 误匹配发票号码（纯数字 20 位无法区分）
   - `amount` / `tax` 合计行解析为 0（正则未匹配到正确行）
   - `invoiceItem` 包含多余字段内容
   - 修复后可恢复快速路径，单张识别总耗时从 ~8s 降至 ~1.2s

2. **大模型 API Key 硬编码**
   - `ARK_API_KEY` 目前写在 `volcengine.ts` 中，应迁移至 `.env`

3. **错误重试机制**
   - 大模型偶发超时/错误时，缺少自动重试逻辑
   - 前端可增加"重新识别"按钮

4. **前端导出功能**
   - 历史记录导出为 Excel（已有 UI 入口，后端接口待实现）

5. **成品油发票特殊字段**
   - `oilQuantity`（数量）、`oilUnitPrice`（单价）字段已预留，识别逻辑待实现

6. **多页 PDF 支持**
   - 当前每次请求只处理单文件，多页 PDF 超 16 页需要分页处理（OCRPdf 单次最多 16 页）

## 十、关键技术决策记录

| 时间 | 决策 | 原因 |
|------|------|------|
| 初期 | 使用 `MultiLanguageOCR` + 本地正则 | 快，但识别准确率差，税号/购销方错误率高 |
| v2 | 引入 Ark 大模型做语义解析 | 解决识别准确率，代价是速度慢 |
| v2.1 | 切换至 `OCRPdf` 智能文档解析 | 返回带版面的 Markdown，大幅提升 OCR 质量 |
| v2.2 | 大模型从 `doubao-seed-1-6-flash` → `doubao-1-5-pro-32k` → `doubao-seed-2-0-mini` → `deepseek-v3-2-251201` | 前三个模型存在大量 reasoning_tokens（内部推理），导致耗时 10-15s；DeepSeek-V3 无 reasoning，稳定在 6-8s |
| v3 | 实现混合策略框架（正则快速路径 + 大模型兜底） | 目标：正则成功时 ~1.2s，失败时降级大模型 ~8s |
| v3.1（当前） | 暂时禁用正则快速路径 | 正则在税号/金额上存在准确率问题，等待修复 |
