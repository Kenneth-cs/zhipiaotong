# 智票通 (SmartInvoice) Web端 产品需求文档 (PRD)

## 1. 项目概述

### 1.1 项目名称
智票通（精准发票管理系统）Web端

### 1.2 项目背景
企业财务人员在日常工作中需要处理大量电子发票（PDF、图片等格式），手动录入发票信息、核对金额、排查重复报销等工作耗时耗力且易出错。本项目旨在开发一款基于Web的智能发票管理工具，通过接入火山引擎AI的OCR识别技术，实现发票的批量自动识别、智能去重检测，并支持一键导出Excel统计报表，极大提升财务人员的工作效率。

### 1.3 核心目标
- **高效录入**：支持批量拖拽上传PDF、JPG、PNG格式发票，自动进行OCR信息提取。
- **智能防重**：系统自动比对发票号码等关键信息，精准标记重复发票，防止重复报销。
- **便捷导出**：一键生成结构化的Excel发票统计台账。
- **历史追溯**：提供发票历史存档与检索功能，方便随时查阅和管理。

---

## 2. 用户角色
- **财务人员/普通员工**：上传发票、查看识别结果、处理重复项、导出报表、管理历史发票记录。

## 3. 功能架构

```text
智票通 Web端
├── 全局导航与布局
│   ├── 左侧菜单（工作台、发票历史、上传发票快捷按钮、设置、帮助）
│   └── 顶部导航（面包屑、全局搜索、消息通知、个人中心）
├── 工作台 (核心业务流)
│   ├── 步骤一：上传发票 (拖拽/点击上传，支持PDF/JPG/PNG)
│   ├── 步骤二：文件预处理 (上传进度展示、待确认列表)
│   ├── 步骤三：统计及去重 (OCR识别、结果展示、重复项标红、总计金额计算)
│   └── 步骤四：下载统计报表 (生成并下载Excel文件)
└── 发票历史 (数据管理)
    ├── 筛选与搜索 (关键词、日期范围、状态、高级筛选)
    ├── 批量操作 (导出选中、删除选中)
    ├── 历史列表 (发票详情、状态展示)
    └── 存储容量展示
```

## 4. 页面与功能详细说明

### 4.1 全局布局与公共组件
- **左侧侧边栏**：
  - 顶部展示系统Logo及名称“智票通 - 精准发票管理”。
  - 主菜单：包含【工作台】、【发票历史】。
  - 快捷操作：常驻【上传发票】主按钮。
  - 底部菜单：包含【设置】、【帮助】。
  - 在“发票历史”页面底部额外展示【存储容量】进度条。
- **顶部导航栏**：
  - 左侧展示当前页面的面包屑/Tab（如：工作台、发票历史）。
  - 右侧包含：全局搜索框（搜索关键词）、消息通知图标、用户头像。

### 4.2 工作台 (Workspace)
工作台是发票处理的核心页面，采用分步引导式设计（1. 上传PDF发票 -> 2. 统计及去重 -> 3. 下载统计报表）。

#### 4.2.1 状态一：初始上传
- **功能描述**：用户在此区域上传需要处理的发票文件。
- **UI 元素**：
  - 顶部Banner：展示功能简介“在线PDF电子发票统计及去重”。
  - 步骤条：高亮当前步骤“1 上传PDF发票”。
  - 上传区域：支持将文件拖拽至虚线框内，或点击【选择文件】按钮浏览本地文件。
  - 格式支持提示：明确提示支持 JPG, PNG, PDF 格式。

#### 4.2.2 状态二：文件预处理与确认
- **功能描述**：文件上传过程中的状态展示及上传完成后的确认。
- **UI 元素**：
  - **实时处理中**：展示正在上传/预处理的文件的进度条（如 45%, 82%），并显示剩余项数量。
  - **待确认列表**：
    - 字段：序号、文件名、发票号码（暂无）、日期（暂无）、金额（暂无）、税额（暂无）、总额（暂无）、状态、操作。
    - 状态显示为“待确认”（黄色标签）。
    - 操作列提供“删除(X)”按钮，允许用户在识别前移除错误上传的文件。
  - **底部操作栏**：
    - 提示文案：“文件已就绪，共有 X 个新文件等待确认识别”。
    - 按钮：【清空列表】、【确认并开始识别 (X张)】（主按钮）。

#### 4.2.3 状态三：统计及去重（识别结果）
- **功能描述**：调用火山引擎AI API完成OCR识别后，展示结构化数据，并进行查重。
- **UI 元素**：
  - 步骤条：高亮步骤“2 统计及去重”。
  - **识别结果列表**：
    - 字段填充：发票号码、日期、金额、税额、总额。
    - **状态标识**：
      - **已识别**（蓝色）：正常识别且无重复。
      - **发现重复**（红色）：系统检测到与本次或历史库中发票号码相同的发票，高亮警示。
      - **识别中...**（灰色）：正在调用API处理中的状态。
    - 操作列：提供“编辑(铅笔)”、“删除(垃圾桶)”、“更多(...)”操作。允许用户手动修正OCR识别偏差。
  - **底部统计操作栏**：
    - 统计信息：发票总计 X 张，**发现 X 个重复项**（标红），总金额 XXX.XX CNY。
    - 按钮：【重置工作台】（清空当前任务）、【生成 Excel】（主按钮）。

#### 4.2.4 状态四：下载统计报表
- **功能描述**：Excel生成完毕后的下载页面。
- **UI 元素**：
  - 步骤条：高亮步骤“3 下载统计报表”。
  - 成功提示：“生成Excel表格完成”。
  - 统计摘要：“此次共为您统计了 X 张发票文件！”。
  - 文件图标及名称：如“发票统计文件.xlsx”。
  - 按钮：【返回继续操作】（回到初始上传页）、【下载Excel表格】（主按钮，触发浏览器下载）。

### 4.3 发票历史 (Invoice History)
- **功能描述**：查看和管理所有历史处理过的发票记录及审计日志。
- **UI 元素**：
  - **顶部操作区**：
    - 标题及描述：“发票存档 - 查看和管理所有历史处理的发票，包含精确的审计日志。”
    - 批量操作按钮：【导出选中到 Excel】、【删除选中】（红色）。
  - **筛选区**：
    - 搜索框：支持按发票号、销售方、金额等关键词搜索。
    - 日期范围选择器：默认“最近 30 天”。
    - 状态筛选：全部、正常、重复。
    - 【高级筛选】按钮。
  - **历史数据列表**：
    - 包含复选框（用于批量操作）。
    - 字段：ID、处理日期、销售方名称、纳税人识别号、金额、状态、操作。
    - 状态标签：“正常”（蓝色）、“重复”（红色）。
    - 操作列：【查看详情(眼睛图标)】、【下载原件(下载图标)】。
  - **底部分页**：标准的分页控件（上一页、页码、下一页、总条数）。

## 5. 非功能性需求

### 5.1 接口与技术要求
- **OCR识别引擎**：**必须接入火山引擎 AI (Volcengine AI) 的发票识别 API**。
  - 需处理API的鉴权、请求封装及响应解析。
  - 需支持处理火山引擎API返回的置信度，对于低置信度字段在前端给予提示（可选）。
- **去重逻辑**：
  - 基于“发票代码 + 发票号码”的组合进行唯一性校验。
  - 需同时与“本次上传批次内”以及“历史数据库中”的记录进行比对。
- **Excel导出**：前端或后端需支持将JSON数据格式化并导出为 `.xlsx` 格式，需包含表头及规范的格式。

### 5.2 性能要求
- **上传与识别响应**：单张发票OCR识别响应时间应控制在 2-3 秒内。支持并发上传和识别处理。
- **大文件支持**：支持多页PDF解析，单文件大小限制建议为 10MB，单次批量上传数量建议不超过 100 张。

### 5.3 兼容性
- **浏览器支持**：适配主流现代浏览器（Chrome, Edge, Safari, Firefox 最新版本）。
- **响应式设计**：主要面向桌面端（PC）使用场景，需适配 1280px 及以上分辨率屏幕。

## 6. 识别准确性优化方案（V1.1 迭代规划）

### 6.1 背景

经38张发票实测，发现两类典型识别问题：
1. **非全电发票识别有误**：旧式增值税电子普通发票（含成品油版）版面结构与全电发票差异大，现有解析逻辑存在字段提取错误。
2. **购买方/销售方字段对调**：部分发票 OCR 文字行顺序特殊，导致买卖双方身份被互换。

---

### 6.2 方案一：发票类型分类前置 + 分支解析

#### 核心思路

在 OCR 文字提取完成后、字段解析之前，**由系统规则（正则/关键词匹配）自动判断发票类型**，再走对应的解析分支。

> **为何选系统规则而非 AI 判断？**
> OCR 文字已提取完毕，发票类型区分特征明确（是否存在"发票代码"字段、抬头关键词），正则匹配耗时 <1ms、零额外成本、结果确定性强，无需额外调用 AI 接口。

#### 分类规则

```text
步骤1：读取 OCR 提取的原始文字行
   ├── 含"发票代码"字段（8位/10位纯数字）？
   │     ├── 是 → 非全电发票，进入子类型判断
   │     │       ├── 含"成品油" / "升" / "单价（元/升）" → 成品油增值税发票
   │     │       ├── 含"增值税专用发票"            → 增值税专用发票
   │     │       └── 其他                          → 增值税电子普通发票
   │     └── 否 → 全电发票（全面数字化电子发票）→ 走现有解析逻辑

步骤2：按类型调用对应字段映射规则
```

#### 非全电发票专属字段映射

| 字段 | 说明 | 处理方式 |
|------|------|---------|
| 发票代码 | 8位或10位纯数字 | 提取入库，用于去重组合键 |
| 密码区 | 加密字符串，无业务含义 | 忽略，不入库 |
| 成品油数量/单价 | 升数、元/升 | 提取入库（可选扩展字段） |
| 货物名称 | 含 `*` 前缀，如 `*汽油*95号车用汽油(VIB)` | 清洗 `*` 后入库 |

---

### 6.3 方案二：购买方/销售方字段对调修复

#### 问题根因

当前解析逻辑依赖 OCR 文字行的顺序/位置推断购销双方，部分发票文字行顺序异常时会造成对调。

#### 解决方案：解析锚点 + LLM Prompt 双重约束

**措施1：解析层改用标签锚点（优先级最高）**

中国增值税发票（全电和非全电）有法定版式，购买方/销售方标识是强制标注字段，应以标签作为锚点而非依赖行序：

```text
解析规则：
- 匹配到"购买方"/"买方"标签后的内容  → 填入 buyerName / buyerTaxId
- 匹配到"销售方"/"卖方"标签后的内容  → 填入 sellerName / sellerTaxId
- 若两者均未匹配到标签              → 标记为低置信度，前端提示人工复核
```

**措施2：LLM 二次提取时强化 Prompt 约束**

若后续引入大模型做语义提取，Prompt 中须明确：

```
「发票中购买方（买方）与销售方（卖方）有明确的文字标签标注，
请严格以"购买方"/"买方"标签识别买方信息，以"销售方"/"卖方"标签识别卖方信息，
不可依赖位置推断，两者不可混淆。」
```

并在 Few-shot 示例中加入购销方容易混淆的样本及正确答案。

**措施3：前端可疑标记**

当 buyerTaxId 与 sellerTaxId 相同、或任一字段为空时，前端自动标记为 ⚠️ 待复核，提示用户手动确认。

> **关于白名单方案（不采用）**：白名单需按企业维护，面向大量用户时管理成本极高，多租户场景下不可扩展，故不采用。

### 6.4 优先级与实施计划

| 优先级 | 方案 | 改动位置 | 预估工作量 |
|--------|------|---------|-----------|
| P0 | 解析锚点修复购销方对调 | `server/volcengine.ts` 正则解析层 | 小（0.5天） |
| P1 | 发票类型分类前置逻辑 | `server/volcengine.ts` 新增类型判断函数 | 中（1天） |
| P2 | 非全电发票专属字段映射 | `server/volcengine.ts` 新增分支解析 | 中（1天） |
| P3 | 前端低置信度/可疑字段标记 | 前端识别结果列表组件 | 小（0.5天） |
