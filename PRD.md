## 二、发票管理（文件夹功能）

### 2.1 方案选择

**结论：在"发票历史"内合并，新增左侧文件夹面板，不新增独立 Tab。**

理由：
- 发票管理和发票历史是同一数据源，独立 Tab 会造成用户困惑（"去哪里找发票"）
- 左侧面板 + 右侧列表的经典双栏布局，用户学习成本低

## 三、兑换码功能（用量控制）

### 3.1 业务规则

| 用户类型 | 每日识别上限 | 说明 |
|----------|-------------|------|
| 普通用户 | **5 张/天** | 超出需使用兑换码 |
| 兑换码用户 | 5张基础 + 兑换码次数 | 次数可叠加，按天消耗 |
| （预留）VIP用户 | 无限制 | 后期付费套餐 |

**次数计算规则：**
- 每自然日（00:00 重置）独立计算
- 兑换码兑换后，次数永久累加到账户余额，每次识别扣 1 次
- 兑换码一经使用即失效，不可转让

### 3.2 兑换码生成方案

采用**命令行脚本批量生成 + 闲鱼手动分发**模式（初期最简方案，无需开发管理后台）。

#### 兑换码格式

`XCQ-XXXX-XXXX-XXXX`

- 前缀 `XCQ` = 小财犬缩写，便于识别品牌
- 后 12 位：大写字母 + 数字，去掉易混淆字符（`O`、`I`、`L`），随机生成
- 示例：`XCQ-A3B7-K9M2-P4N8`

#### 面值档位（次数）

| 档位 | 次数 | 建议闲鱼售价 |
|------|------|------------|
| 小包 | 10 次 | ¥9.9 |
| 中包 | 50 次 | ¥39 |
| 大包 | 100 次 | ¥69 |

#### 管理员生成流程（方案一：命令行脚本）

```
1. 在项目根目录执行脚本（服务器或本地均可）：
   npx tsx scripts/generate-codes.ts --count=50 --quota=10

2. 脚本自动将兑换码写入数据库（状态：unused），
   并在当前目录输出 codes_20260512.txt，格式如下：
   XCQ-A3B7-K9M2-P4N8
   XCQ-C1D5-F8G3-H2J6
   ...

3. 管理员打开 codes.txt，复制对应兑换码，
   在闲鱼订单完成后手动通过平台消息发给买家。

4. 买家登录小财犬，在工作台输入兑换码 → 点击兑换 → 次数到账。
```

**脚本参数说明：**

| 参数 | 说明 | 示例 |
|------|------|------|
| `--count` | 生成数量 | `--count=50` |
| `--quota` | 每张码的面值（次数） | `--quota=10` |
| `--output` | 输出文件名（可选） | `--output=batch1.txt` |

**脚本位置：** `scripts/generate-codes.ts`（待开发）

### 3.3 用量检测流程

```
用户发起识别请求
       │
       ▼
查询今日已用次数（used_today）
       │
       ├── used_today < 5 → 直接识别，used_today + 1
       │
       └── used_today >= 5 → 查询账户余额（quota_balance）
                  │
                  ├── quota_balance > 0 → 扣减余额，执行识别
                  │
                  └── quota_balance = 0 → 返回 403，提示兑换码或明日再试
```

### 3.4 数据库变更

## 六、识别准确性优化计划（V1.1）

### 6.1 问题一：非全电发票识别有误
旧式增值税电子普通发票（含成品油）版面与全电发票差异大，字段解析出错。

**修复方案：发票类型分类前置 + 分支解析**
系统利用正则规则在提取文字后立即判断类型（如根据是否含"发票代码"或特定关键词），根据类型分支执行不同的解析逻辑（全电发票、成品油发票、普通电子发票等），从而避免将不同格式的发票套用同一套规则。

### 6.2 问题二：购买方/销售方字段对调
部分发票 OCR 行序特殊，导致买卖双方身份被互换。

**修复方案：标签锚点解析**
放弃依赖行序推断，改用文字标签（如“购买方”、“销售方”）锚点进行精准匹配。若遇无标签情况，辅助前端预警提示人工复核。

### 6.3 问题三：数字“1”和“7”容易混淆识别
**问题表现：**
由于发票字体或扫描模糊，金额字段中数字“1”和“7”极易被 OCR 引擎混淆（例如实际税额 `0.47` 错误识别为 `0.41`）。

**修复方案：金额多维度财务交叉验证与自愈纠错**
提取出“金额”、“税额”和“价税合计”后，不完全信任原始结果，在后端加入自动化财务逻辑自愈层：

1. **数学等式穷举校验（核心）**：
   - 根据发票硬逻辑：`金额 + 税额 = 价税合计`。
   - 当校验不通过时，如果金额字段包含 "1" 或 "7"，程序自动尝试互换（1换7，7换1），进行再次求和验证。若替换后的数值使得等式完美成立（如 `47.43 + 0.47 = 47.90`），则将该数值自动纠偏。
2. **法定税率推算（辅助验证）**：
   - 验证纠正后的数字是否符合国内常见发票税率（1%, 3%, 6%, 9%, 13% 等），提高纠错的确信度。
3. **中文大写金额比对（兜底校验）**：
   - 解析发票的“大写金额”（如：肆拾柒圆玖角整），大写汉字几乎不会有 1/7 的混淆。将其转为阿拉伯数字作为“价税合计”的绝对标尺，来反推和校验其他小写金额错误。
