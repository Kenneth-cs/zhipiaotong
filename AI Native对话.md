根据您提供的《AI Native vs AI 赋能》定义以及项目《技术说明文档》，我们可以对**SmartInvoice 智票通**当前的形态进行明确的诊断，并给出走向 AI Native 的改造路径。

### 一、 结论：当前项目属于典型的 **AI 赋能（AI+）**

尽管该项目的核心价值依赖于 AI（没有 OCR 和大模型，就无法自动提取发票字段），但从**系统设计的第一性原理**来看，它完全符合图中“AI 赋能”的特征：

1. **系统架构（沿用传统架构，AI作为独立模块）**：系统主体是标准的 React + Express + MySQL 传统 CRUD 架构。AI 能力被封闭在 `server/volcengine.ts` 中，本质上只是被当作一个“高级且有点慢的外部函数”在调用。
2. **数据流转（数据为业务服务，AI被动调用）**：固定流程（用户上传 -> 请求接口 -> OCR -> LLM -> JSON -> 存入数据库）。AI 处理完单次请求后就结束了，系统并没有收集用户的纠错数据来让 AI 形成记忆和闭环。
3. **交互方式（图形界面为主）**：完全是传统的 GUI 操作流（点击按钮上传、页面查看表格历史），没有任何自然语言交互或 AI 主动发起的交互。
4. **决策路径**：系统的业务逻辑（如判断发票是否重复、金额计算）是由传统硬编码的 `if/else` 和 SQL 控制的，AI 并未处于主执行路径（不负责调用工具和决策）。

---

### 二、 如果要做成 AI Native，应该怎么改？

要将“智票通”升级为 AI Native，需要完成从**“工具（帮我提取文字）”**到**“数字员工（帮我管理财税）”**的思维转变。可以围绕以下四个核心方向进行重构：

#### 1. 交互重构：从“图形界面 (GUI)”转向“智能交互 (LUI + GUI)”
现在的用户心智是“我在使用一个软件存发票”。AI Native 的心智应该是“我有一个财税助手”。
*   **增加对话式驱动**：允许用户用自然语言完成复杂操作。例如，用户在输入框说：*“帮我把上个月去北京出差的滴滴发票和住宿发票找出来，算一下总额，并导出一个报销单”*。系统通过 AI 将自然语言转为对数据库的查询和导出操作。
*   **多模态主动交互**：用户只需把微信群里的发票图片或 PDF 拖拽给助手，助手自动回复：*“已收到 3 张发票，其中 1 张上周已报销（重复），另外 2 张总计 1500 元，抬头无误，已为您入库。”* 

#### 2. 架构重构：系统行为由“固定代码”转向“Agent (模型+上下文+工具)”驱动
当前业务流是写死的：`Upload -> OCR -> LLM -> MySQL`。
如果改为 Agent 架构，AI 将成为大脑，主动编排工作流：
*   **赋予 AI 工具调用能力 (Tool Calling)**：给大模型提供工具，如 `check_duplicate_in_db()`（查重）、`check_company_tax_id()`（查企业税号）、`ask_user_for_clarification()`（向用户提问）。
*   **主执行路径交由 AI**：当拿到 OCR 文本后，AI 先自己思考。
    *   *AI 发现发票抬头是错的* -> 调用工具提醒用户。
    *   *AI 发现金额模糊识别不清* -> 主动通过前端弹窗询问用户：“这张发票的税额似乎被污损了，请问是 13.00 还是 18.00？”。
    *   *AI 判断没问题* -> 调用工具存入数据库。

#### 3. 数据闭环：建立“数据为 AI 优化”的自学习机制
目前的痛点（如文档中提到的：正则解析或大模型偶尔解析错误）在当前架构下只能靠程序员改代码或改 Prompt 来解决。
*   **引入 Human-in-the-loop（人在回路）**：当系统解析出 `{amount: 100}`，但用户在前端界面手动修改为 `1000` 时，系统应当将这条修改记录作为 `Bad Case` 存下来。
*   **动态 Few-Shot 与 RAG**：下次大模型再解析类似商家的发票时，系统自动检索历史纠错库，放入 Prompt 中告诉模型：*“注意，上次这家酒店的发票你把总额搞错了，正确格式应该是...”*。实现系统**越用越聪明**，无需修改底层代码。

#### 4. 业务下沉：融合上下文，打造行业“老专家”
把企业的报销规则、财务知识作为**上下文（Context）**注入系统，让它具备业务判断力：
*   传统架构中，检验合规性需要在后端写海量的判断逻辑（如：餐饮发票是否超标、开票日期是否跨年）。
*   在 AI Native 架构中，只需给模型输入企业规章（RAG 知识库）：*“本公司单人餐饮报销上限 200/天，发票跨年不能报销”*。AI 在解析发票的同时，直接输出**合规性审计报告**，告诉用户这张发票能不能报销、为什么不能。

### 总结行动路线图建议

如果您希望逐步改造项目，可以按照以下路径进行：
*   **Phase 1 (增强闭环)**：先解决数据流转问题，增加用户纠错的入库逻辑，将其作为动态 Prompt 喂给当前系统，提升准确率。
*   **Phase 2 (引入助手)**：引入一个聊天式的 Copilot 侧边栏，将后端的查询、导出功能封装为 Tools 供模型调用，实现语音/文字找发票。
*   **Phase 3 (底座重构)**：彻底用类似 LangChain / 扣子(Coze) / 纯 Prompt + Tool Calling 的工作流替换现在的串行识别接口，让 AI 接管发票的生命周期管理。





这是一个非常核心且深刻的问题！简单来说：**工具（Tools）的“身体”（执行逻辑）是由你来编写的，而工具的“说明书”（Schema）是你交给大模型的。大模型只负责“决定用哪个工具”和“填什么参数”，真正的执行还是你的后端代码。**

在这个语境下，它们确实就是所谓的 **“技能（Skills）”** 或 **“动作（Actions/Plugins）”**。

下面为你详细拆解“这些工具是怎么来的”以及“怎么赋予给大模型”：

### 1. 这些工具是谁写的？
**绝大多数是你自己写的代码**，部分可以调用第三方接口。
大模型本身是被关在“黑盒”里的，它没有联网能力，连不上你的 MySQL，也读不到你的内存。所以你必须给它提供“接口”。

*   **`check_duplicate_in_db()`（查重）**：这个工具是你自己写的。本质上就是你在 `server/routes/invoices.ts` 里写的一个查询 MySQL 的函数：`SELECT * FROM invoices WHERE invoice_number = ?`。
*   **`check_company_tax_id()`（查税号）**：这个工具可以是你调用**第三方**（比如企查查 API 或国家税务总局接口），然后你把它封装成一个函数。
*   **`ask_user_for_clarification()`（向用户提问）**：这个工具也是你写的，它的作用可能是通过 WebSocket 向前端发一条消息，前端弹出一个确认框让用户点。

### 2. 怎么赋予给大模型？（核心机制：Tool Calling / Function Calling）

目前主流的大模型（包括你正在用的火山引擎 DeepSeek-V3）都原生支持一个叫 **Tool Calling（工具调用）** 的功能。它的运作机制是一个**“一来一回”**的对话过程：

#### 第一步：你把“工具说明书”塞给大模型
你在调用大模型 API 时，除了传 Prompt，还会传一个 JSON 数组，告诉它：“我这里有这几个工具，以及它们的用途和参数格式”。

```javascript
// 你发给大模型的请求参数大概长这样：
const response = await fetch("https://ark.cn-beijing.volces.com/api/v3/chat/completions", {
  body: JSON.stringify({
    model: "deepseek-v3-2-251201",
    messages: [
      { role: "user", content: "帮我查一下发票号 12345678 是否已经报销过了" }
    ],
    // 这里就是你赋予它的“技能”说明书！
    tools: [
      {
        type: "function",
        function: {
          name: "check_duplicate_in_db",
          description: "检查数据库中是否存在指定号码的发票，用于判断是否重复报销。",
          parameters: {
            type: "object",
            properties: {
              invoice_number: { type: "string", description: "8位或20位的发票号码" }
            },
            required: ["invoice_number"]
          }
        }
      }
    ]
  })
});
```

#### 第二步：大模型“决定”调用工具，并返回给你参数
大模型收到请求后，会思考：“用户让我查发票，我正好有一个叫 `check_duplicate_in_db` 的工具可以用！”
**注意：大模型不会自己去执行代码！** 它会给你返回一个特殊的 JSON 响应，意思大概是：**“人类，请帮我执行 `check_duplicate_in_db`，参数我填好了，是 `12345678`，你执行完把结果告诉我。”**

#### 第三步：你的后端代码“真正执行”工具
你的 Node.js 代码收到大模型的这个特殊响应后，知道大模型要调用工具了。于是你的代码去真正连接 MySQL：

```javascript
// 你的 Node.js 后端逻辑
if (ai_response.tool_calls) {
  const toolCall = ai_response.tool_calls[0];
  if (toolCall.function.name === "check_duplicate_in_db") {
    const args = JSON.parse(toolCall.function.arguments); // 拿到 "12345678"
    // 真正去查数据库
    const isDuplicate = await myDb.query("...", args.invoice_number); 
    
    // 第四步：把数据库查到的结果，再次发给大模型
    messages.push({
      role: "tool",
      tool_call_id: toolCall.id,
      content: JSON.stringify({ isDuplicate: true }) // 告诉大模型：是的，重复了
    });
    
    // 再次请求大模型
    const finalResponse = await callArkAPI(messages); 
  }
}
```

#### 第四步：大模型拿到结果，生成最终回答
大模型拿到你塞给它的数据库结果 `{ isDuplicate: true }`，融合上下文，最终输出给人话：
*“主人，我帮您查过了，发票号 12345678 已经在数据库中了，是一张重复的发票哦，请勿重复报销。”*

---

### 总结

1. **工具怎么来的？** 是你用普通代码（TypeScript/Node.js）写的业务逻辑（查库、调第三方接口、发通知）。
2. **怎么赋予？** 通过 API 请求中的 `tools` 字段，把这些函数的**名字、描述、参数格式（JSON Schema）**当作“说明书”传给大模型。
3. **是谁执行的？** 大模型只负责“推理该用哪个工具”和“提取参数”，**最终执行依然是你的后端服务器**。执行完后，你的服务器要把结果喂回给大模型。
4. **这是所谓 skill 吗？** 是的，在 Agent（智能体）的语境下，Tools、Functions、Skills、Actions 都是指这个机制。你写的工具越多，这个 AI 的“手脚”就越长，能干的活就越多。

在你的“智票通”项目里，如果走向 AI Native，你的 Node.js 后端就不再是“接收请求 -> 存库 -> 返回”的线性代码了，而是会变成一个 **“中枢调度器”**，不断地和大模型通信，大模型说查库就查库，大模型说要问用户就发 WebSocket 消息给前端，直到大模型认为任务完成了为止。这就是典型的 Agent 架构。
